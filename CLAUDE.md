# CLAUDE.md тАФ AI Agent Context for drt

This file gives AI agents (Claude Code, Cursor, etc.) the context needed to work effectively in this codebase.

## What is drt?

**drt** (data reverse tool) is a CLI tool that syncs data from a data warehouse (BigQuery) to external services via declarative YAML configuration. Think of it as the reverse of dlt: `dlt` loads data *into* a DWH; `drt` activates data *out of* a DWH.

**Tagline:** "Reverse ETL for the code-first data stack."

## Architecture

```
Config Parser тЖТ Source (BigQuery) тЖТ Sync Engine тЖТ Destination (REST API)
                                                         тЖУ
                                                   State Manager
```

Key design principle: **module boundaries are drawn for future Rust rewrite (PyO3)**. The `engine/sync.py` module is the primary Rust candidate тАФ keep it pure (no I/O side effects beyond protocol calls). Logging, state persistence, OTel spans, and any other observability/persistence side effect MUST flow through `drt.engine.observer.SyncObserver`. Direct `logger.*`, `state_manager.save_sync(...)`, or `watermark_storage.save(...)` calls inside `engine/sync.py` are guarded by `tests/unit/test_engine_observer.py` boundary checks and will fail CI.

## Package Layout

```
drt/
тФЬтФАтФА cli/          # Typer CLI commands
тФЬтФАтФА config/       # Pydantic models + YAML parser
тФЬтФАтФА connectors/   # Connector registry тАФ auto-discovery of sources/destinations
тФЬтФАтФА sources/      # Source Protocol + BigQuery impl
тФЬтФАтФА destinations/ # Destination Protocol + REST API impl
тФЬтФАтФА engine/       # Sync orchestration (future Rust core)
тФЬтФАтФА state/        # Local JSON state persistence
тФФтФАтФА templates/    # Jinja2 renderer (future MiniJinja/Rust)
```

## Protocols (critical interfaces)

- `Source.extract(query: str, config: ProfileConfig) -> Iterator[dict]`
- `Destination.load(records: list[dict], config: DestinationConfig, sync_options: SyncOptions) -> SyncResult`
- `StateManager.get_last_sync / save_sync`

Connector dispatch uses a centralized registry (`drt/connectors/registry.py`) тАФ adding a new connector requires registering it there, not editing `main.py`. Implementations use `assert isinstance(config, SpecificConfig)` for type narrowing. `type: ignore` is only allowed for external library issues.

## What NOT to do

- Do not add a GUI or web UI тАФ this is a CLI-first tool
- Do not add RBAC or multi-tenancy тАФ small team / personal use
- Do not add `type: ignore` тАФ only allowed for external library issues (`no-untyped-call`, `import-untyped`)
- Do not add heavy dependencies to core тАФ extras (`[bigquery]`, `[mcp]`) exist for a reason
